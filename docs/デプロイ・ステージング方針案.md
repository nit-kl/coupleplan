# デプロイ・ステージング方針（P0-3）

最終更新: 2026-04-25

## 目的

`P0-3` の論点である「どこにデプロイし、どうステージング運用するか」を、MVP向けに比較して決定しやすくする。

## 前提

- 現在は技術選定（`T1`〜`T3`）が未確定。
- 先に「運用の型」を決め、実装技術に依存しにくい方針を採る。
- 予算は小さく、運用コストを最小化する。

## 要件（MVP）

- ステージング環境と本番環境の2環境を持つ
- `main` 反映で本番デプロイできる
- 環境変数を環境ごとに分離できる
- 失敗時にロールバックしやすい
- ログ確認が容易

## 候補比較

| 案 | 概要 | 長所 | 注意点 |
|----|------|------|--------|
| A | Vercel（Web）+ Supabase（DB/Auth）+ GitHub Actions | 初速が最速、プレビューが簡単、無料枠で開始しやすい | ベンダーロックイン、ジョブ制御は設計要 |
| B | Cloudflare Pages/Workers + D1/Queues | エッジ寄りで高速、コスト効率がよい | 学習コスト、アプリ構成に制約が出やすい |
| C | Render/Fly.io + Neon + GitHub Actions | 柔軟性が高くAPI構成に寄せやすい | 運用の手間が増える可能性 |

## 採用方針（MVP）

**案B（Cloudflare Pages/Workers + D1/Queues）** を採用する。  
決定記録: [ADR 0001](./adr/0001-deployment-platform-cloudflare.md)

- エッジ配信と運用コストのバランスが良い
- MVPで必要な構成（Web/API/DB/ジョブ）をCloudflare基盤で揃えやすい
- 2環境（Staging/Production）でシンプルに運用できる

## ステージング運用ルール案

| 環境 | 用途 | 更新トリガー | データ |
|------|------|--------------|--------|
| Staging | 受け入れテスト | `develop` ブランチへのマージ時 | 本番相当の匿名化データ |
| Production | 本番提供 | `main` ブランチへのマージ時 | 本番データ |

## 最小リリースフロー案

1. feature ブランチで開発
2. PR 上で `npm run lint` をCIで通過
3. レビュー後 `develop` へマージ
4. `develop` マージを契機に Staging へ反映
5. Staging で確認後、`main` へマージ
6. `main` マージを契機に Production へ反映

## P0-3 の決定事項

- デプロイ基盤: 案B（Cloudflare）を採用
- 環境構成: `Staging / Production` の2環境
- 反映トリガー: `develop` マージで Staging、`main` マージで Production
- Secrets 管理: Cloudflare 側シークレット管理を正とし、CIは必要最小限のみ保持
- ロールバック方針: 直前の安定版リリースへ戻す運用を採用（詳細手順は `P0-6` で整備）
