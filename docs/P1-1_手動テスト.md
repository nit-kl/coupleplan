# P1-1 手動テスト手順（会員・ペア連携）

最終更新: 2026-04-30

## 前提

- **カップル招待は2アカウント**前提。同一ブラウザで連続利用する場合は、プロファイル分離（シークレットウィンドウ等）を推奨（セッション混線防止）。
- API: `npm run api:dev`（既定 `http://127.0.0.1:8787`）。F/E から `VITE_API_BASE_URL`（[.env.example](../apps/web/.env.example)）で接続先を上書き可。
- UI: `npm run web:dev`（`http://localhost:5173`）
- プロトタイプ [prototype/index.html](../prototype/index.html) は**要件の参照のみ**。本受け入れの操作対象は `apps/web`。
- 自動検証: `npm run api:smoke`（**AとBの2人分**＋`POST /auth/otp/*` 〜 招待受諾まで。カップルは `active`・メンバー2人を**必須**に検証）。CI では `npm run ci:pr` に同梱。Staging Worker を叩く例: 環境変数 `API_BASE_URL=https://<worker>.workers.dev`（平文 `debugCode` が返る `ENVIRONMENT=staging` 前提。本番は手動受け入れのみ想定）
- **Cloudflare 上で P1-1 完走**する手順は下記 [Cloudflare Staging/本番（Git 連携）](#cloudflare-staging本番git-連携) を正とする。

## Cloudflare Staging/本番（Git 連携）

GitHub の `.github` にデプロイ用 YAML を足さず、**Cloudflare ダッシュボードでリポジトリを接続**して Worker / Pages をデプロイする想定。二重デプロイを防ぐため、従来の `deploy-api.yml` は廃止している。

### 1. D1 データベースと `wrangler.toml`

1. [Cloudflare ダッシュボード](https://dash.cloudflare.com)にログインし、Wrangler 用の API トークン（または `wrangler login`）を用意する。
2. 例（ローカル CLI）: `cd apps/api` のうえで  
   `npx wrangler d1 create coupleplan-staging` / `npx wrangler d1 create coupleplan` など、[wrangler.toml](../apps/api/wrangler.toml) の `database_name` に対応する DB を作る。
3. 出力された **UUID** を `wrangler.toml` の該当 `database_id` に貼る（`00000000-0000-…` のままではリモートにバインドできない）。
4. リモートへ初回マイグレーション: リポジトリルートで  
   `npm run d1:migrate:staging` / `npm run d1:migrate:production`（[0001_initial.sql](../apps/api/migrations/0001_initial.sql)）。

### 2. Resend（メール OTP 送信）の設定

[Resend](https://resend.com) は、Worker から [HTTPS API](https://resend.com/docs/api-reference/emails/send-email) 経由でトランザクションメールを送る。本リポの実装は [apps/api/src/infra/email/otpEmail.ts](../apps/api/src/infra/email/otpEmail.ts)（`RESEND_API_KEY` / 任意 `RESEND_FROM`）。

**既に Resend を使っている場合（ドメイン検証・API キーあり）**

- **Domains** に検証済み（Verified）のドメイン（例: `coupleplan.com`）があれば、新規の DNS 作業は不要。ダッシュボード上の**リージョン**（例: Tokyo）は Resend 側のデータ所在の指定であり、**Worker のコードは変更不要**（送信は API キーで統一される）。
- **API Keys** では、**Sending access** 付きの既存キーをそのまま `RESEND_API_KEY` に使ってもよい。Supabase 等と**別キーにしたい**・漏えい時の切り分けのため、**+ Create API Key** で `coupleplan-cf-workers` など Worker 専用を新規発行し、そのトークンだけを Cloudflare の Secret に入れるのが無難（既存キーは**再表示できない**ため、中身が分からなければ新規発行でよい）。
- **`RESEND_FROM`**: 検証済みドメイン上のアドレスを使う。例: `CouplePlan <otp@coupleplan.com>` や `CouplePlan <noreply@coupleplan.com>`（自ドメインの方針に合わせる。角括弧内のメールは Resend の該当ドメイン配下に存在する扱いでよいことが多い）。

上記のどれにも当てはまらない場合は、手順 1 から順に進める。

1. **アカウント作成**  
   [resend.com](https://resend.com) にサインアップ（GitHub 連携可）。未登録のときだけ必要。

2. **API キー**  
   ダッシュボード → **API Keys** → **+ Create API Key**。名前（例: `coupleplan-cf-workers`）を付与し、権限は **Sending access**（送信だけで足りる）。表示された `re_` で始まるキーを**一度だけ**コピー。  
   - この値を Cloudflare Worker の **Secret** `RESEND_API_KEY` として入れる（手順は下記 **§3** の変数表）。**リポジトリにコミットしない。**

3. **送信元（From）を決める**（`RESEND_FROM`）

   | 方式 | 内容 | 向き |
   |------|------|------|
   | 検証用（すぐ試す） | Resend 既定の `onboarding@resend.dev` など。無料枠のルール内で、**受信可能な本人宛**に限定されやすい。 | 初回の到達確認 |
   | 本番向け | **Domains** で自分のドメインを追加し、向け先に出る **DNS レコード**（多くは SPF 等）を DNS に登録。検証が **Verified** になったら `CouplePlan <otp@example.com>` のように指定。 | 任意の相手のメール宛 |

   - コード未設定時の既定は実装上 `CouplePlan <onboarding@resend.dev>`（[createOtpEmailFromEnv](../apps/api/src/infra/email/otpEmail.ts)）。`RESEND_FROM` を **Plaintext 変数**で上書き可（形式は `表示名 <メール@ドメイン>`。角括弧内は検証済みドメインの送信可能アドレス）。
   - **サンドボックス/無料**では宛先に制限があることがある。届かないときは [Resend のドキュメント](https://resend.com/docs)とダッシュボードのドメイン状態を確認。

4. **疎通のしかた**  
   - Worker に `RESEND_API_KEY` を入れたあと、ブラウザから OTP 申請。受信箱に `CouplePlan ログインコード` 件名のメールが届く（本文は平文＋有効時間の目安。実装文面は上記 TypeScript 参照）。  
   - 失敗時は Worker の **Logs** で 5xx や `email_unavailable`（送信 API 失敗）を確認。

5. **本リポの環境変数との対応（再掲）**

   | 名前 | 種別 | 役割 |
   |------|------|------|
   | `RESEND_API_KEY` | Secret | Resend API。必須（メール配信を有効にする） |
   | `RESEND_FROM` | 変数（平文可） | 未設定なら上記既定。独自ドメイン送信用。 |

### 3. API（Workers）の接続

1. **Workers** → プロジェクト作成 → **Git に接続**（[Workers Builds ドキュメント](https://developers.cloudflare.com/workers/ci-cd/builds/)に従う）。
2. ウィザード **「Set up your application / アプリを設定」** では、次表のとおり**このリポジトリ（ルートに `package.json` と [wrangler.toml](../apps/api/wrangler.toml) は `apps/api` 下）**向けの値例を入れる。UI ラベル名は英語/日本語表記の差あり。

   | 項目 | 推奨する入力値 | 補足 |
   |------|----------------|------|
   | リポジトリ | `nit-kl/coupleplan` 等、接続した GitHub リポジトリ | 接続直後の画面で確定。 |
   | プロジェクト名 | `coupleplan-api` または Staging 用に `coupleplan-api-staging` | ダッシュボード上の表示名。[wrangler.toml](../apps/api/wrangler.toml) の `name` / `[env.staging] name` と揃えると取り違いにくい。 |
   | ビルド コマンド | `npm ci` | **推奨。** リポジトリ**ルート**で実行。ルート [package.json](../package.json) により `wrangler` 等の依存を取得する。空でもデプロイだけ動くことはあるが、CI と揃えるなら入れる。 |
   | デプロイ コマンド | `npx wrangler deploy --config apps/api/wrangler.toml --env staging` | 非本番（`develop` 等）向け。**本番**用の本番接続を別プロジェクト/別ブランチで切るなら `--env production`。 |
   | 非本番ブランチのビルド | **オン** | `main` 以外のブランチでプレビュー/Staging 用ビルドを有効化。 |
   | 詳細設定（非本番のデプロイ コマンド） | 上表の「デプロイ コマンド」と同じコマンドを入れる。 | 初期表示の `wrangler versions upload` は、バージョン管理・カナリア用。Staging にそのまま `deploy` するなら、メインのデプロイ コマンドと揃えてよい。 |
   | 詳細設定 → パス | `/` | **リポジトリのルート**（`package.json` がある階層）を指す。`apps/api` だけにパスを切ると、ルートの `npm install` 結果を参照できず扱いづらいため、`/` + `--config apps/api/wrangler.toml` の組を推奨。 |

3. **API トークン**  
   - **新しいトークンを作成する**を選び、**トークン名**（例: `coupleplan-workers-build`）を入力しておく。  
   - 表示どおり、デプロイ用トークンはここで自動生成される。既存の **Edit Cloudflare Workers** 権限付きトークンを使う場合は、該当トークンを選ぶ。  

4. **変数 / Environment variables**（デプロイ前またはデプロイ後 **Settings → Variables and Secrets** でも可）

   | 変数名 | 例・形式 | 暗号化 |
   |--------|----------|--------|
   | `RESEND_API_KEY` | Resend ダッシュボードの API キー `re_...` | **Encrypt オン**（シークレット扱い） |
   | `RESEND_FROM` | （任意）`CouplePlan <onboarding@あなたのドメイン>` または使用ドメインに合わせる | プレーンテキスト可 |
   | `ALLOWED_ORIGINS` | 例: `https://<pages-project>.pages.dev,http://localhost:5173` | プレーンテキスト。Pages の**実オリジン**を入れないとブラウザで CORS 失敗。 |

5. 右下 **デプロイ** を押して初回デプロイ。完了後、表示される **Worker URL**（`https://<...>.workers.dev`）を控え、下記 **§4 フロント（Pages）** の `VITE_API_BASE_URL` に使う。  

6. **Settings → Variables and Secrets** で、上表（手順 4）に足りない変数を後追い可。`wrangler.toml` の `[env.staging.vars]` の `ENVIRONMENT` は toml 側で定義済み。必要ならダッシュボードの vars で上書き可。

### 4. フロント（Pages）

1. **Workers & Pages** → **Pages** → **Create a project** → 同じリポジトリを **Connect to Git**。
2. ビルド設定は、リポジトリルート基準で次のとおりが目安（UI 名はプロダクト更新で異なる場合あり）。

   | 項目 | 推奨する入力値 | 補足 |
   |------|----------------|------|
   | プロジェクト名 | `coupleplan-web` 等 | Workers 側の名前と区別しやすくする。 |
   | 制作フレームワークのプリセット | Vite 等、ない場合はなし | 任意。 |
   | ルート ディレクトリ / Path | `/` または空 | ルートの [package.json](../package.json) で `npm run web:build` する想定。 |
   | ビルド コマンド | `npm ci && npm run web:build` | ルートから実行（[vite.config.ts](../apps/web/vite.config.ts) は `apps/web` を root に指定済み）。 |
   | ビルド出力ディレクトリ | `dist/web` | Vite の `outDir` に合わせる。 |
   | 環境変数（ビルド時） | 下表 | **必須。** 未設定だとフロントは `localhost:8787` の API を指す。 |

   | 変数名 | 値の例 | 補足 |
   |--------|--------|------|
   | `VITE_API_BASE_URL` | `https://<Worker名>.<アカウント>.workers.dev` | §3 でデプロイした **Staging Worker の URL**（末尾 `/` なし）。本番 Pages なら Production Worker の URL。 |

3. 保存・初回デプロイ後、割り当てられた **Pages の URL**（`https://*.pages.dev`）を控え、§3 の Worker の **Variables** にある `ALLOWED_ORIGINS` にそのオリジンを**追記**して再デプロイする（先に Pages を作っていないと URL が分からない場合は、一度仮で `https://placeholder.pages.dev` を入れず、**Pages デプロイ後に Worker 側だけ更新**してもよい）。
4. プレビュー/本番ブランチはチーム方針に合わせる（`develop` / `main` 等）。

### 5. メール OTP のワンパス受け入れ

1. ブラウザで **Pages** の URL を開き、会員申請のメールアドレスで OTP を受け取る（Resend 側のドメイン/宛先制限に注意）。
2. 別メール/別プロファイルで **下記シナリオ A** を実施（デバッグ用に、Staging では API 応答に `debugCode` が含まれることもあるが、**本手順の正は受信メール**）。

### 6. 参考コマンド（ローカルから）

- Staging へ手動デプロイ: `npm run api:deploy:staging`
- リモート Staging 向けスモーク: `API_BASE_URL=https://...workers.dev npm run api:smoke`（Node の fetch なので CORS は掛からない。`debugCode` 必須のため Staging 向け）
- テストで作った2ユーザーの掃除（Staging）: `npm run p1:cleanup:testdata:staging -- user1@example.com user2@example.com`
- テストで作った2ユーザーの掃除（Production）: `npm run p1:cleanup:testdata:production -- user1@example.com user2@example.com`

### 7. テストデータ削除（2ユーザー）

手動テストで作成した2人分のデータを、D1 からまとめて削除するための補助コマンドを用意している。

1. 削除対象の2メールアドレスを決める（A-1/A-2 で使ったもの）。
2. 環境に応じて次を実行する。
   - Staging: `npm run p1:cleanup:testdata:staging -- user1@example.com user2@example.com`
   - Production: `npm run p1:cleanup:testdata:production -- user1@example.com user2@example.com`
3. 必要なら同じメールで再度シナリオAを実行し、初回登録としてやり直せることを確認する。

削除対象は、指定メールに紐づく `users` / `sessions` / `otp_requests` / `auth_audit` と、関係する `couples` / `couple_members` / `invites`。

## シナリオA（必須）: 2端末/2プロファイルで正規フロー

### A-1. 先に登録する人（招待コードを出す）

1. 起動画面で **「先に登録（招待コードを出す側）」** を押す
2. メール・表示名を入力 → **次へ（ペア連携）**
3. 表示された `CP-` コードを控える、または **コードをコピー**
4. このまま、相手の参加待ち案内の文言が読めることを確認する

### A-2. あとから参加する人（別ブラウザまたはシークレット）

1. 新しいウィンドウ/プロファイルで同じURLを開く
2. **「あとから参加（コードを受け取った側）」** を押す
3. **A-1 とは違うメール**でプロフィール入力 → **次へ**
4. **招待コード**欄に A-1 の `CP-` を入力 → **カップルに参加する**
5. 登録完了画面に遷移する

### A-3. 先に登録した人（招待者の確認）

1. A-1 の画面に戻る
2. **相手が参加したか確認する** を押す
3. 登録完了画面（または有効化済みを示す状態）に至る

**期待:**

- 途中で、招待された側に「自分の `CP-` コード表示」は**不要**（被招待専用UIに切り替わる）
- 招待者がコード未入力のまま「完了」扱いに**ならない**（旧仕様の抜けを防ぐ目的）
- 最終的に `GET /couples/me` 相当の結果が、カップル `active` かつメンバー2人になる（スモークと整合）

## シナリオB: 再発行

1. 招待者でペア連携画面まで進む
2. **コードを再発行** → 表示コードが `CP-` 形式の**別**の値に変わる

---

## シナリオC: 招待者が参加を急かされるケース

1. 被招待者が、誤ったコードを入れる
2. 期待どおり **エラー**（HTTP 4xx / アラート）になる
3. 正しいコードで再実行できる

（メッセージ文言は `apps/api` の `AppError` 依存）

---

## シナリオD: API 未起動

1. API を止めた状態で「次へ（ペア連携）」

**期待:** 接続失敗が通知される（`Failed to fetch` 等）

---

## 判定基準

- シナリオ **A**（2人分）が期待どおり
- 可能なら B/C/D も通過
- 若しくは `npm run api:smoke` 成功 ＋ A の主要操作を目視
