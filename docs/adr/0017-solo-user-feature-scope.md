# 0017. ソロ利用フェーズにおける機能境界（案 A）

## ステータス

承認済み

## コンテキスト

- 公開初期のファネルでは、カップル前提のオンボーディングが重く、**認証（OTP）まで到達しない・ペア成立まで進まない**ケースが問題になっている。
- 一方でデートルーレット・サイレント・ニンジャのコアは **二人の合意やカップル単位の状態**に依存する（[0011](./0011-date-roulette-state.md)、[0012](./0012-silent-ninja-declarations.md)、[0013](./0013-ninja-weekly-aggregation.md)）。
- **大幅なデータモデル変更なし**に摩擦を下げるため、「まず一人で入り、ペアは後から」という **案 A** を採用する必要がある。

## 決定

### 用語: ソロ利用フェーズ

**ソロ利用フェーズ**とは、ユーザーが **メール OTP により認証済み**であるが、次のいずれかに該当する状態を指す。

1. どのカップルにも所属していない（例: `GET /couples/me` がカップルを返さない）。
2. 所属カップルが **`pending`** であり、**二人のメンバーが揃っていない**（招待側のみ・相手未参加など。[0010](./0010-accept-invite-from-pending-solo-member.md) の誤操作リカバリとも整合）。

**ペア成立済み**とは、上記に当てはまらず、カップルが **`active` かつ二人のメンバーが揃っている**状態とする（API・DB の実際の判定は実装に委ねるが、プロダクト上の「コア体験開始条件」はこれに合わせる）。

### プロダクト境界（案 A）

ソロ利用フェーズでは次を **許可**する。

- **ホーム（ダッシュボード）**の表示、およびペア成立に向けた **案内・オンボーディング**。
- **ペア連携**（招待コードの発行・入力、招待の受諾に至る導線）。
- **プロフィール／アカウント系**（表示名変更、ログアウト、退会など既存仕様）。

ソロ利用フェーズでは、次の **「二人で完走する体験」は開始しない**。

- **デートルーレット**: セッションに基づく進行（スワイプ〜マッチ〜抽選〜結果確定までの一連）。※一覧の閲覧のみ許すか非表示にするかは UI 実装の詳細とし、本 ADR の境界は「体験としての完走を開始しない」こと。
- **サイレント・ニンジャ**: 申告・週次集計に基づく「体験としての完走」。※任務一覧の閲覧のみ許すかは UI 詳細。

ソロ利用フェーズから該当機能へ遷移しようとした場合は、**ペア成立への誘導**（および必要なら短文説明）とし、コア API を無断で連打できないようにする（既存の 404 / 業務エラーと整合させる）。

### 明示的にやらないこと（本 ADR のスコープ外）

- ソロのみでルーレット・ニンジャを **二人分と同等に完走**させる仕様は定義しない（別判断・別 ADR）。
- LP・計測・コピー自体は [開発計画書 Phase G](../開発計画書.md) で扱う。

## 結果

### メリット

- 認証とペア連携に集中でき、獲得ファネルと整合する。
- コアのカップル依存モデル（0011〜0013）と矛盾しにくい。

### トレードオフ

- 「ソロでも一部だけ触れる」案より入口の魅力は限定的になりうる。必要なら LP・チュートリアルで補う。
- `pending` 単独と「完全にカップル無し」の **UI の差**（同じソロ扱いか）を揃えると実装が単純になる。
