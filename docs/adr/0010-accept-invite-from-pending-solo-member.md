# 0010. pending単独メンバーは招待参加時に409にしない

## ステータス

承認済み

## コンテキスト

- 既存実装では、`acceptInvite` が「カップル所属あり」を一律 `409 already_in_couple` としていた。
- そのため、被招待者が誤って先に「招待コードを出す側」を踏み、`pending` の1人カップルを作ると、正規の招待参加に戻れなくなる。
- MVPでは誤操作からの自己回復導線を優先し、退会や手動DB掃除なしで参加完了できる必要がある。

## 決定

- `acceptInvite` の事前チェックを次のように変更する。
  - 既存所属が `active` の場合は従来どおり `409 already_in_couple`。
  - 既存所属が「`pending` かつ本人1名のみ」の場合は、その仮カップル（関連 `invites` 含む）を削除してから招待参加を続行する。
- 上記削除処理はリポジトリ層に明示メソッドとして実装し、`D1` / `InMemory` で同等挙動を保証する。

## 結果

### メリット

- 誤って先行登録した被招待者でも、同一セッションのまま招待参加に復帰できる。
- 「カップル成立前はコンフリクトを過度に起こさない」UXになる。

### トレードオフ

- `pending` 状態の分岐が増え、状態遷移テストの拡充が必要になる。
- 仮カップル削除の副作用（招待コード失効）を仕様として明示し続ける必要がある。
