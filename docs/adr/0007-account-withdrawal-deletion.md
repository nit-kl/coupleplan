# 0007. 退会時はカップル単位で物理削除する

## ステータス

承認済み

## コンテキスト

- `P1-1` の会員・ペア連携フローに、ユーザー自身で退会できる導線を追加する。
- 既存のコアモデルは `Couple` を中心に `User`, `CoupleMember`, `Invite` を扱う（[0003](./0003-core-entities-couple-invite.md)）。
- MVPでは、共有データを相手側に残すよりも「退会したらふたりの検証データを確実に消す」ことを優先する。

## 決定

- 認証済みユーザーの退会は `DELETE /users/me` で実行する。
- 退会対象ユーザーがカップルに所属している場合、そのカップルに属する全メンバーの `users`, `sessions`, `couple_members`, `invites`, `couples` を物理削除する。
- 退会対象ユーザーがカップルに所属していない場合は、そのユーザーの `users` と `sessions` を物理削除する。
- ルーレット/ニンジャなど後続テーブルを追加する場合も、カップルに属する共有データは同じ削除境界へ含める。

## 結果

### メリット

- MVP検証で「退会＝全データ削除」と説明しやすく、ユーザーの期待に合う。
- 相手側に中途半端なカップル状態や招待コードを残さない。
- D1の外部キー `ON DELETE CASCADE` と整合し、実装を小さく保てる。

### トレードオフ

- 片方の退会で相手の共有データも削除されるため、プロダクト内で確認文言を明確にする必要がある。
- 将来、履歴保持や復元要件が出た場合は、匿名化や論理削除へ設計変更する必要がある。
