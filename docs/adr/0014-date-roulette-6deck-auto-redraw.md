# 0014. デートルーレットはセッション6枚デッキ・1件交差成立・0件時自動再抽出

## ステータス

承認済み

## コンテキスト

- 既存のデートルーレットは静的カタログ全件（12〜15件）を毎回対象にし、双方 like 交差 3 件以上で成立（[0011](./0011-date-roulette-state.md)）としていた。
- 実運用観点では「毎回ラインナップが同じ」ことによる新鮮味の不足が課題で、順番よりも候補集合そのものの変化が重要である。
- 1 ラウンドで全件を評価する負担もあり、体感的には 6 件程度で完了する短いサイクルが望ましい。

## 決定

### セッションデッキ

- セッション作成時に、静的カタログから重複なしで **6 件**を抽出し、その `plan_id` 集合をセッションに保存する。
- `/roulette/plans` は「グローバルな全件」ではなく、**現在 active なセッションの6件デッキ**を返す。
- 投票の妥当性検証は、既存の `findPlanById` に加えて **当該セッションデッキ内か**を必須条件にする。

### マッチ成立条件

- 両者がそのセッションデッキの全件を評価し終えた時点で、双方 like の交差数を評価する。
- 交差が **1件以上**で `ready` へ遷移し、`spin` 可能とする。

### 0件時の自動再抽出

- 交差が 0 件の場合、ユーザー操作を挟まずに現セッションをアーカイブし、新しい 6 件デッキのセッションへ自動遷移する。
- これにより「何もマッチしなかった時はラインナップを変えて再投票」を標準フローにする。

## 結果

### メリット

- 毎回の候補集合が変わるため、体験のマンネリ化を抑制できる。
- 6件デッキにより投票完了までの負荷が下がり、回転率が上がる。
- 1件成立により失敗ラウンドが減り、ルーレット演出に到達しやすい。

### トレードオフ

- セッションにデッキ情報を持つため、永続化スキーマの更新が必要。
- 0件時に自動で新セッションへ切り替わるため、クライアントはセッション更新に追従してデッキ再取得する必要がある。
