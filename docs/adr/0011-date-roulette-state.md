# 0011. デートルーレットは非同期投票・3件交差・サーバー乱択で確定する

## ステータス

置換（[0014](./0014-date-roulette-6deck-auto-redraw.md)）

## コンテキスト

- 開発計画書 §3 / §4.3 / Phase 1 の P1-2・P1-3 で、デートルーレットの「プラン参照・本人投票・マッチ候補計算」と「抽選・結果保存」を実装する必要がある。
- 企画書 §4.1 の体験フロー（スワイプ → ブラインドマッチ → ルーレット演出 → 結果）を MVP の最小構成で再現したい。
- カップルの2ユーザーは別端末・別タイミングでアクセスする想定で、リアルタイム同期は MVP ではコストに見合わない。
- 一方で、相手のいいね/パスは結果が出るまで隠す（ブラインド投票）必要がある。

## 決定

### セッション

- カップル単位で `RouletteSession` を 1 件だけ "active" として保持し、状態は `collecting → ready → decided` の前進のみ。
- `decided` のセッションは凍結し、再抽選不可。再挑戦は新セッションを開始（過去セッションはアーカイブ＝以後の `getOrCreateActiveRouletteSession` の対象外）。
- カップルが `active` でないユーザーは利用不可（API は `412 couple_required` を返す）。

### プランカタログ

- MVP のプラン候補は **API バンドルの静的 TS 定数**（`apps/api/src/data/plans.ts`）として管理し、12〜15 件をコードレビューで運用する。
- D1 にカタログテーブルは作らない。`roulette_votes.plan_id` の参照整合性は **アプリ層**（`findPlanById`）で検証する。
- 将来 CMS 化する場合は同型の `PlanCard` を D1 テーブルへ移しても usecase の改修だけで済む形にする。

### 投票（非同期・ブラインド）

- 各ユーザーは自分のタイミングで全プランをスワイプし、`POST /roulette/sessions/me/votes` で **一括 upsert**（`{ planId, vote: like|pass }[]`）する。
- 同一 `(session_id, user_id, plan_id)` は UNIQUE 制約で 1 行に維持し、再投票は上書き。
- セッション取得 API は collecting 中、相手の vote 内容（like/pass）を返さず、**「投票完了したか」と「自分の投票数」のみ**を返す。
- 両ユーザーがそれぞれ全プランを評価し終えた時点で交差集合（双方 like）を計算し、**3 件以上**で `ready` へ遷移する。3 件未満は `collecting` のまま、UI は再投票（やり直し）を促す。

### 抽選

- `ready` のセッションでのみ `POST /roulette/sessions/me/spin` を許可。
- サーバー側で `crypto.getRandomValues` を用いた一様乱択により候補から 1 件を選び、`roulette_results`（1 セッション 1 件 UNIQUE）に保存し `decided` へ遷移する。
- すでに `decided` の場合は **既存結果を冪等に返す**（再抽選しない）。`spin` も `restart` も同じ振る舞いに揃える。

### 再挑戦

- `POST /roulette/sessions/me/restart` で `decided` セッションをアーカイブし、新規 `collecting` を作成する。`collecting` / `ready` 中の `restart` も同様に「現セッションを破棄し新規作成」を許す（誤投票時の救済）。

### MVP 外

- 写真証拠・地域/予算/カテゴリフィルタ・カスタムプラン・SNS 自動投稿・OGP 画像・PWA Push 通知・履歴 UI。
- リアルタイム同期は持たず、相手の進捗反映は **UI 側のポーリング**で吸収する（数秒間隔・上限数回）。

## 結果

### メリット

- 二人の都合に合わせて遅延参加できるため、MVP のリリース体験を阻害しない。
- 静的カタログ + アプリ層検証でデータ層の表現を最小化し、実装と運用を軽量に保てる。
- サーバー乱択により結果の一貫性・冪等性を担保しやすい（端末間ズレが起きない）。

### トレードオフ

- ポーリング前提のため、相手の進捗反映に最大数秒の遅延がある。
- カタログ更新が API デプロイに紐づくため、頻繁な編集は不向き（運用上は本格 CMS まで小回り重視で進める）。
- 投票テーブルの `plan_id` は外部キー制約を持たない。アプリ層検証を確実に維持する必要がある。
