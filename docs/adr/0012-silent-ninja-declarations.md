# 0012. サイレント・ニンジャの任務カタログと自己申告（相手非通知）

## ステータス

承認済み

> 更新注記: 任務カタログ方針は [0016](./0016-ninja-custom-missions-with-guardrails.md) により部分更新（静的のみ -> 制約付きカスタム任務を許可）。

## コンテキスト

- 開発計画書 Phase1 の P1-4 で、ニンジャの「任務参照・申告」を MVP 範囲で実装する。
- 企画書 §4.2 では、家事などを行い **アプリ内でニンジャポイントを自己申告**し、**相手に即時は通知しない**（秘密性）ことがコア体験である。
- デートルーレット同様、任務マスタの CMS は MVP 外とし、静的カタログで十分な運用負荷にしたい（[0011](./0011-date-roulette-state.md) と同型）。

## 決定

### 任務カタログ

- MVP の任務一覧は **API バンドルの静的 TS 定数**（`apps/api/src/data/ninjaMissions.ts`）とする。ポイントは **サーバー側の定義のみ**有効とし、クライアントから点数を送らせない（改ざん防止の最低限）。
- D1 に任務マスタテーブルは置かない。`mission_id` の参照整合性は **アプリ層**（カタログ検索）で検証する。

### 申告ログ

- `ninja_logs` に 1 申告 1 行を保存する。列は `mission_id` と当時の `point`（監査用にログ時点の点数を保持）、`created_at`（UTC ISO）。
- 同一任務の **再申告は許可**する（週内に複数回できる）。制限は β 以降で検討する。

### 相手非通知（個票）

- API は **原則として本人の申告一覧のみ**を返す。相手の **個別の申告行（どの任務をいつやったか）** は、週次公開後も MVP では返さない（「サイレント」の維持）。
- 週次集計で **合計ポイントのみ**を公開する（[0013](./0013-ninja-weekly-aggregation.md)）。

### 認可

- `active` なカップル（メンバー 2 人）が前提。それ以外は `412 couple_required`（既存のルーレットと同様）。

## 結果

### メリット

- 静的カタログでデータ層と運用を軽く保てる。
- 点数のサーバー確定により最低限の公平性を担保できる。
- 個票を伏せたまま合計だけ週次で出せる。

### トレードオフ

- 任務追加・調整はデプロイが必要（本格 CMS は後続）。
- 虚偽申告の抑止はポリシー説明と β 以降の設計に委ねる（開発計画書 §8）。
